# Outreach — Signal Dictionary

**Category:** Sales engagement
**Role in stack:** AE/SDR sequence engine for outbound + multi-thread nurture; system of record for prospect-level engagement events (delivered, clicked, replied, opted-out, bounced) and the only place we see the *cadence* of buying-committee contact at email granularity.
**Integration surface:** REST API v2 conformant to [JSON:API 1.0](https://jsonapi.org/format/1.0/) ([overview](https://developers.outreach.io/api/)), webhooks ([reference](https://developers.outreach.io/api/webhooks/)), OAuth 2.0 Authorization Code grant ([flow](https://developers.outreach.io/api/oauth/)) or S2S for backend-only apps ([s2s](https://developers.outreach.io/api/s2s-access)). Base host: `https://api.outreach.io/api/v2/`.
**Pricing/access reality:** API access requires an Outreach plan tier that includes API (Professional+) and an OAuth app registered in the customer's instance with scopes selected per resource. Webhook registration requires `webhooks.all.read` and `webhooks.all.write`. Access tokens TTL ~7200s; refresh tokens persist and must be stored securely (per [oauth flow doc](https://developers.outreach.io/api/oauth/)). Bulk reads available via the [bulk API](https://developers.outreach.io/api/bulk-api).

## What it emits
Per [API reference](https://developers.outreach.io/api/reference/), the resources we care about are `prospects`, `mailings`, `sequenceStates`, `sequences`, `sequenceSteps`, `mailboxes`, `accounts`, `opportunities`, and `opportunityProspectRoles`. Each follows JSON:API shape (`data.type`, `data.id`, `data.attributes`, `data.relationships.<rel>.data.{type,id}`). Engagement is captured both as counters on `prospects` (`openCount`, `clickCount`, `replyCount`, `bouncedAt`, `optedOut`, `optedOutAt`) and as per-send rows on `mailings` (one row per Sequence × SequenceStep × Prospect send, with state transitions `delivered → opened → clicked → replied → bounced` carrying timestamps `deliveredAt`, `openedAt`, `clickedAt`, `repliedAt`, `bouncedAt`).

The [webhooks reference](https://developers.outreach.io/api/webhooks/) confirms subscribable resources: `account`, `call`, `emailAddress`, `import`, `kaiaRecording`, `mailing`, `opportunity`, `opportunityProspectRole`, `prospect`, `sequence`, `sequenceState`, `task`, `user`. Mailings specifically emit `mailing.bounced`, `mailing.delivered`, `mailing.opened`, `mailing.replied` in addition to `created/updated/destroyed`. Sequences emit `advanced` and `finished`. This is enough to compute per-persona engagement deltas in near-real-time without polling — but see "Hardest part" below for the delivery-SLA gotcha.

## Signals we'd extract

### 1. Champion reply latency decay — BLOCKING
- **What it is:** The contact mapped to `opportunityProspectRoles` with `role` ∈ {`Economic Buyer`, `Champion`} on an opportunity with `stage='Selected Vendor'` stops replying to sequenced mail at their historical cadence.
- **Why for the wedge:** Champion going dark between verbal commit and procurement is THE Selected Vendor failure mode. The signal lives in `mailings.repliedAt` deltas 1–3 weeks before SFDC `Opportunity.LastActivityDate` stales out enough to look anomalous.
- **Rule shape:** For `prospects` joined to champion contacts on Selected Vendor opps: if rolling 14d median `(mailings.repliedAt - mailings.deliveredAt)` > 5d AND prior 60d baseline of the same metric < 24h AND no `mailing.replied` webhook fired in last 7d for any prospect on that opp → fire.
- **Source fields / endpoints:** `GET /mailings?filter[prospect][id]=<id>&filter[state]=replied&sort=-repliedAt` ([Mailing tag](https://developers.outreach.io/api/reference/tag/Mailing/)); `data.attributes.repliedAt`, `data.attributes.deliveredAt`, `data.relationships.prospect.data.id`, `data.relationships.sequenceStep.data.id`. Join key: `prospects.attributes.emails[]` → SFDC `Contact.Email` → `OpportunityContactRole`.

### 2. Buying-committee opt-out / bounce — BLOCKING
- **What it is:** Any contact attached to the opp's `opportunityProspectRoles` (Finance, Legal, IT, Security, Procurement, Champion) flips `prospects.attributes.optedOut` false→true, gets `optedOutAt` populated, or has `bouncedAt` newly set on an active Selected Vendor opp.
- **Why for the wedge:** Opt-outs from Finance/IT after they were looped in mean the deal review is over and the rep doesn't know yet. Bounces on Procurement = wrong contact, restart needed before quarter-end.
- **Rule shape:** Subscribe to `prospect.updated` webhook with `payloadVersion: 2`. Per [webhooks doc](https://developers.outreach.io/api/webhooks/), v2 includes a `beforeUpdate` object — diff `data.attributes.optedOut` against `meta.beforeUpdate.optedOut` (false→true) OR `data.attributes.bouncedAt` (null→timestamp). If matched prospect's email is in any open Selected Vendor opp's `opportunityProspectRoles` → BLOCKING Slack DM to AE owner within 5 min. Fallback poller (every 60s) on `GET /prospects?filter[updatedAt]=<since>&fields[prospect]=optedOut,optedOutAt,bouncedAt,emails` to backstop the no-retry webhook policy (see Hardest part).
- **Source fields:** `data.attributes.optedOut`, `data.attributes.optedOutAt`, `data.attributes.bouncedAt`, `data.attributes.emails`, `data.attributes.emailsOptedOut`, webhook header `Outreach-Webhook-Signature` for HMAC-SHA256 verify.

### 3. Multi-thread activation at the account — ACTION
- **What it is:** A net-new `prospect` at the same `accounts.id` (not previously on any OCR) hits `mailing.replied` or `mailing.clicked` on an active sequence while opp is in Selected Vendor — typically the deputy, exec sponsor, or new committee member surfacing.
- **Why for the wedge:** Expansion of the buying committee mid-deal is a *positive* signal worth instrumenting — the rep should add them to the OCR immediately and re-mirror collateral. Often the actual decision-maker reveals themselves here.
- **Rule shape:** On `mailing.replied` or `mailing.clicked` webhook ([Mailing events](https://developers.outreach.io/api/reference/tag/Mailing/)): resolve `data.relationships.prospect.data.id` → `GET /prospects/{id}?include=account` → check `account.id` against open Selected Vendor opps. If prospect.email NOT in current OCR AND `prospects.attributes.title` regex matches `(VP|Director|Chief|Head of) (Finance|Legal|IT|Security|Ops|Procurement)` → ACTION task: "Add [name] to OCR + send committee primer." Use `sequenceStates.attributes.state ∈ {active, paused}` filter to ignore replies on finished sequences.
- **Source fields:** `prospects.attributes.title`, `prospects.attributes.emails`, `prospects.relationships.account.data.id`, `mailings.attributes.state`, `sequenceStates.attributes.state` ([SequenceState tag](https://developers.outreach.io/api/reference/tag/SequenceState/)).

## What we'd ignore
- `mailing.opened` events outright — Apple MPP and corporate link-scanners (Mimecast, Proofpoint) prefetch images and rewrite links, so `openedAt` and even initial `clickedAt` are noise. Only count `clickedAt` when the click is >10s post-deliver AND not the first click in the link's history.
- Bot clicks: filter `mailings` where `clickedAt - deliveredAt < 2000ms` (gateway prefetch tell).
- SDR-prospected leads not yet linked to an `opportunity` via `opportunityProspectRoles`.
- `sequenceState.created` and `sequence.finished` events without an engagement delta — enrollment ≠ buyer signal.
- Internal/test prospects (`prospects.attributes.emails[]` domain matches the workspace's own corporate domain) and replies from a `mailboxes.attributes.email` we own.
- Calls and tasks resources for v1 — voice/call data is Gong's job; we don't double-source.

## Effort to wire
- **Adapter LOC estimate:** ~400 LOC TS — OAuth 2.0 Authorization Code handler with refresh-token rotation, webhook receiver with `Outreach-Webhook-Signature` HMAC-SHA256 verify, three resource pollers (`prospects`, `mailings`, `sequenceStates`) using `filter[updatedAt]` cursors, JSON:API relationship hydration helper (one-level `include=` resolver), bulk-API fallback for backfill.
- **Time estimate:** 1.5 days — half a day on OAuth + webhook registration round-trip (`POST /webhooks` with `payloadVersion: 2`), half on the SFDC OCR join (the load-bearing piece), half on rule tuning against the last 90d of historical `mailings`.
- **Hardest part (two-part):**
  1. **Webhook delivery has no retries.** Per [webhooks doc](https://developers.outreach.io/api/webhooks/): "No retries on any status codes (including 500, 429)", 4 network-failure retries at 1s intervals, 5s response timeout, no redirect-following. For BLOCKING-tier signals we cannot rely on webhooks alone — the 60s reconciliation poller on `filter[updatedAt]` is mandatory, not optional.
  2. **Joining `prospects.emails[]` → SFDC `Contact.Id` → `OpportunityContactRole` reliably.** Reps create prospects with personal-domain aliases, contacts get merged in SFDC, and OCR is notoriously under-maintained. Without a clean join — ideally via an Outreach custom field carrying `Contact.Id` (see Install-time discovery) — every signal in this card misfires.

## Install-time discovery
- **Custom Prospect fields:** Outreach exposes `custom1`–`custom150` on `prospects`. We need to know (a) whether any are already mapped to SFDC `Contact.Id` / `Account.Id` via the SFDC plugin, (b) which slot we can claim if not, and (c) whether any custom field encodes persona/committee role (so we don't have to title-regex). Pull `GET /prospects?fields[prospect]=custom1,custom2,...` against a known seed contact to inventory.
- **Sequence IDs to filter to:** Get the list of `sequences` actively used for late-stage / champion-nurture motions — `GET /sequences?filter[enabled]=true&fields[sequence]=name,shareType,tags`. We only want to score engagement on sequences tagged for post-demo / Selected Vendor nurture, not top-of-funnel cold outbound (which has its own baselines and would corrupt the latency-decay model in Signal 1).
- **Mailbox-domain allowlist:** `GET /mailboxes?fields[mailbox]=email,sendDisabled,userId` to enumerate the workspace's own mailbox domain(s) (and any acquired-brand domains). Required to (a) filter out internal-reply noise in Signal 3 and (b) detect rep-side `sendDisabled=true` mailboxes, which would silently zero out a rep's outbound and look like Signal 1 firing on every account they own.
- **OCR enforcement at Selected Vendor gate:** Confirm whether RevOps blocks stage progression without populated `opportunityProspectRoles`. If yes, the email→Contact→OCR join is high-confidence and Signal 2 can be BLOCKING. If no, we drop Signal 2 to ACTION and add account-domain + title-regex as a heuristic fallback.
- **Gong/Chorus reply double-count:** Confirm whether the Gong-Outreach integration writes inbound replies back as `mailings` rows (which would inflate `replyCount`) or only attaches them to the SFDC activity timeline. If the former, dedupe by `mailings.attributes.bodyHtml` hash + `repliedAt` ±60s window before computing latency.
